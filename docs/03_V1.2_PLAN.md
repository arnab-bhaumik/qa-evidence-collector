# QA Evidence Collector — V1.2 Plan: Jira Integration

## Overview

V1.2 adds Jira integration to QA Evidence Collector. After generating a DOCX report, the tester can upload it directly to a Jira issue as an attachment — without leaving the app. Jira connection details are stored securely in the Settings dialog.

---

## Motivation

QA teams typically log test evidence against Jira issues (bug reports, test execution tickets, etc.). Currently testers must manually open Jira, navigate to the issue, and attach the file. V1.2 removes that friction — one click uploads the report directly to the correct Jira issue.

---

## New User Workflow (V1.2)

1. User opens **Settings** → configures Jira URL, Project Key, Email, and API Token
2. User clicks **Test Connection** — confirms credentials are valid
3. User saves settings
4. After generating a report, user clicks **Upload to Jira**
5. A dialog asks for the **Jira Issue Key** (e.g. `QA-123`)
6. Report is uploaded as an attachment to that Jira issue
7. Success message with a link to the issue

## S1 — Jira Configuration Section Detail

A new **Jira Configuration** section is added at the bottom of the Settings dialog.

### Fields

| Field | Type | Description |
|-------|------|-------------|
| Jira URL | Text input | Base URL of the Jira instance, e.g. `https://yourcompany.atlassian.net` |
| Project Key | Text input | Jira project key, e.g. `QA` |
| Email | Text input | Atlassian account email used for API auth |
| API Token | Password input (masked) | Atlassian API token (generated at id.atlassian.com) |

### UI Design
- Section header: **"Jira Configuration"** with a divider line
- Four labelled input fields in a form layout
- API Token field uses `QLineEdit` with `EchoMode.Password` (masked)
- **Save** button saves all settings including Jira config

---

## S2 — Test Connection Detail

A **Test Connection** button sits below the Jira config fields.

### Behaviour
1. Reads URL, Email, and API Token from the fields
2. Makes a `GET /rest/api/3/myself` request to the Jira API
3. Shows an inline status label:
   - Green: **"Connected — logged in as [display name]"**
   - Red: **"Connection failed — check your URL, email, or API token"**
4. Does not save config — just validates

### API Call
```
GET {jira_url}/rest/api/3/myself
Authorization: Basic base64(email:api_token)
```

## S3 — Output Options Checkboxes Detail

A new **Output** section replaces the current output folder row in Settings.

### Checkboxes

| Checkbox | Default | Description |
|----------|---------|-------------|
| Save to folder | Checked | Saves the DOCX to the local output folder (existing behaviour) |
| Upload to Jira | Unchecked | Uploads the DOCX to a Jira issue after generation |

- At least one must be checked (validation)
- Both can be checked — saves locally AND uploads to Jira
- Checkbox state persisted to config

## S5 — Upload to Jira Button + Issue Key Dialog Detail

### Toolbar Button
- New **Upload to Jira** button added to the toolbar (visible only when Jira is configured and "Upload to Jira" checkbox is enabled)
- Enabled only when a report has been generated in the current session

### Issue Key Dialog
- Simple modal dialog with a single text input
- Placeholder: `e.g. QA-123`
- Validates format: must match `[A-Z]+-[0-9]+`
- **Upload** and **Cancel** buttons

### Flow
1. User clicks **Upload to Jira**
2. Issue Key dialog opens
3. User enters issue key and clicks **Upload**
4. Progress indicator shown (button disabled, label "Uploading...")
5. On success: `QMessageBox` — "Uploaded successfully! View issue: {url}"
6. On failure: `QMessageBox` — error message with details

## Security Note

The Jira API token is stored in plain text in the local app config file (`settings.json`). This is acceptable for a single-user desktop tool. The config file is stored in the user's local app data folder and is not transmitted anywhere other than to the configured Jira instance.

---

## Out of Scope for V1.2

- Creating new Jira issues from within the app
- Searching / browsing Jira issues
- Jira Server / Data Center support (Cloud only for now)
- OAuth authentication
- Attaching individual screenshots (only the final DOCX report)
